Документ для **поочерёдного** разбора модулей (логика, UI/стили, ошибки, мёртвый код, дубли, доступы).  
**Правка:** не удалять готовые секции — только дополнять. Статусы обновлять в шапке каждой вкладки.

**Объём кода (ориентир):** `dashboard.js` ~1.0k, `production.js` ~1.4k, `inventory.js` ~1.7k, `purchases.js` ~0.7k, `recipes.js` ~0.8k, `references.js` ~0.3k, `equipment.js` ~0.3k, `docs_registry.js` ~0.3k, `admin.js` ~0.2k, `salary.js` ~1.5k, `sales.js` ~3.8k, `finance.js` ~4.3k строк.

## 1. Дашборд (`dashboard-mod`)

- **Статус:** ✅
- **Навигация:** у всех пользователей, пункт виден.
- **Ключевые файлы:** `views/modules/dashboard.ejs`, `public/js/dashboard.js`, бэкенд: в основном `routes/finance.js` (`/api/analytics/cost-constructor`, `/api/analytics/dashboard-widgets`, настройки DDS, MRP-агрегаты), плюс `routes/inventory.js` (оценка склада), `routes/production.js` (MRP summary в кнопке статуса).

### Логика (клиент + данные)

- Конструктор себестоимости: период `dashPeriodType` (день/неделя/месяц/квартал/год/всё) + расчёт `startDate`/`endDate` локально, затем `POST /api/analytics/cost-constructor` — тот же **паттерн дат**, что и в финансах, но **дублированный** код, не shared-модуль.
- Виджеты (`loadDashboardWidgets`) — `GET /api/analytics/dashboard-widgets` (поступления, прочие KPI в одном ответе).
- Капитализация склада: `loadStockValuation` → `/api/inventory/valuation`.
- MRP: `openMrpPanel` / `checkMrpStatus` — `/api/production/mrp-summary`, обновление кнопки в шапке.
- `initDashboard` (при `switchModule` на дашборд, см. `scripts.ejs`) программно **кликает** `loadCostConstructor` через `querySelector('button[onclick="loadCostConstructor()"]')` — хрупко при смене разметки.
- `loadCostConstructor` кэширует категории в `ccAllCategories` (однократно) через `GET /api/categories`.
- Массовые правки категорий/оверрайды транзакций — сильная связь с **финансовым** справочником, не «изолированный» дашборд.

### API / бэкенд

- Тяжёлая аналитика сосредоточена в `routes/finance.js` (сложные SQL/агрегаты). Риск: изменение определения «цикла» или групп расходов затрагивает и отчёты, и дашборд — нужна единая документация формул.
- «Ожидаемые поступления» на дашборде — **свой** расчёт в `dashboard-widgets` (согласовывать с бизнес-правилом: долг по заказам vs непривязанные платежи; частично пересекается с обсуждением `allocateClientAdvance`).

### Стили и UI

- Сетка `summary-grid`, `card`, аккордеоны вручную через `getElementById` + `classList.toggle` (как в других модулях).
- Кнопка **«Матрица статей»** в `dashboard.ejs` с классом `hidden` — в интерфейсе **не видна**; функция `openCategoryMatrix` в JS **жива** (вызывается из `updateCategoryGroup` после сохранения группы).

### Ошибки и риски

- Фоновая проверка MRP: ошибки в `try/catch` + `console.error` — для пользователя только смена текста кнопки, без явного toста при части сценариев.
- `initDashboard` форсит пересчёт через клик по `onclick` — при правке шаблона дашборд может **не** загрузить конструктор при первом входе.

### Мёртвый код

- Скрытая кнопка матрицы в EJS — **функция не мёртвая**, **кнопка** фактически убрана с UI: либо вернуть в интерфейс, либо вынести в меню/настройки.

### Дублирование

- Расчёт границ периода **скопирован** из идей «финансового» календаря (`finance.js` / `renderFinPeriodUI` vs `renderDashPeriodUI` в `dashboard.js`).
- `fmtRub` локально в дашборде; похожее форматирование есть в `Utils.formatMoney` в других модулях.

### Итог и приоритеты

- **P0:** согласовать с бизнесом определения виджетов (поступления vs финансы/ЗК) и зафиксировать в `erp_technical_docs` или комменте к `dashboard-widgets`.
- **P1:** вынести общий «период в даты» + единая кнопка/портал открытия матрицы категорий без дублирования.

## 2. Финансы (`fin-mod`)

- **Статус:** ✅
- **Навигация:** `admin-only` (пункт в сайдбаре).
- **Ключевые файлы:** `views/modules/finance.ejs`, `public/js/finance.js`, `routes/finance.js`, `utils/allocateClientAdvance.js`.

### Логика (клиент + данные)

- Крупнейший JS-модуль: операции, счета, графики Chart.js, налоги, касса, справочник ДДС, импорст/переводы, карточка контрагента, `reconcileAdvances` (ручная сверка авансов).
- Период и фильтры состояния на клиенте; `loadFinanceData` после сокета `finance_updated` (см. `core.js`).

### API / бэкенд

- Единая точка `routes/finance.js` (очень объёмна): транзакции, план, отчёты, аналитика, настройки. `allocateUnlinkedClientIncome` при вставке транзакции и эндпоинт `reconcile-advances`.
- **Роль `admin` на пункте меню** не означает, что **все** API финансов требуют admin — детализация прав по эндпоинтам при ужесточении RBAC.

### Стили и UI

- Часть разметки собирается строками в JS; крупные inline-`onclick` в `finance.ejs` (аккордеон прибыльности).
- Согласованность с `--*` в целом соблюдается, но **плотность** UI и **длина** файла усложняют сопровождение.

### Ошибки и риски

- Редкие `fetch` вручную (например, профиль контрагента) с явным `Authorization` — остальное через `API.*` (единообразие в целом лучше, чем в Sales).

### Мёртвый код

- В таком объёме **вероятны** устаревшие ветки; без coverage целенаправленно не вычищалось. Рекомендация: поиск `function` / веток с `if (false)` / закомментированных блоков отдельным PR.

### Дублирование

- Логика периода, TomSelect, модалки — пересечения с дашбордом, продажами, закупками.
- `erpCategories` / категории — SSoT на `window.erpCategories` (хорошо), но дубли вызовов `loadFinanceCategories` стоит контролировать.

### Итог и приоритеты

- **P0:** стратегия уменьшения размера `finance.js` (подмодули или разбивка по «операции / настройки / аналитика»).
- **P1:** явная матрица «эндпоинт — кто видит (admin/все)» при расширении ролей.

- MRP-данные и кнопки в **дашборде** и **производстве** — двойная точка входа к одной аналитике.

- Материалы, контрагенты, счета, дневной журнал закупок, поиск, быстрый поставщик. Загрузка по `switchModule('purchase-mod')` → `loadPurchaseMaterials` (`scripts.ejs`).

- Все пути **через** `/api/inventory/purchase*` — не отдельный `routes/purchase.js` (когнитивно важно для разработчика).

### Мёртвый код

- `window.loadPurchaseMaterials` в конце файла — для совместимости; ок.

### Дублирование

- Список материалов/контрагентов дублирует загрузки с **инвентаризацией** и **производством** — кэш `cache.js` / общие загрузчики смотрить при оптимизации.

### Итог и приоритеты

- **P2:** единая функция `loadItemsForSelect` на клиенте с кэшем TTL.
- **P3:** переименование файла `purchases.js` ↔ модуль `purchase` в документации (уже отмечено в мастер-листе).

## 6. Продажи (`sales-mod`)

- **Статус:** ✅
- **Навигация:** у всех.
- **Ключевые файлы:** `views/modules/sales.ejs`, `public/js/sales.js`, `routes/sales.js`.

### Логика (клиент + данные)

- Заказы, корзина, отгрузки, история, спецификации, договора, взаимодействие с контрагентами и балансами. `initSales` на сокет `sales_updated`.
- **Много** прямых вызовов `fetch('/api/...')` **без** `API.get/post` и **без** видимого заголовка `Authorization` (первые 50+ строк: счета, настройки, контрагенты). В `web.js` защита `app.use('/api', authenticateToken)` — **теоретически** ответ 401, если токен не передаётся иным способом. **Нужна проверка в браузере** (успешен ли `loadSalesAccounts`): при 401 селекты пусты, ошибка только в `console`. Это **главный технический риск** модуля.

### API / бэкенд

- `routes/sales.js` + часть контрагентов/договоров из других роутеров. Аналитика `GET /api/sales/analytics` и т.д.

### Стили и UI

- Табы, большие списки; согласованность в целом, но **объём** файла мешает review.

### Ошибки и риски

- Несогласованность `fetch` vs `API` → разное поведение при 401/403 (где-то toasts из `core`, где-то нет).
- Глобаль `window.FINANCE_TAX_PERCENT` из `loadFinanceTaxPercent` — **скрытая** связь с налоговыми настройками.

### Мёртвый код

- Аналогично finance: низковесный поиск `grep` на неиспользуемые `function` **рекомендован** отдельно (файл слишком велик).

### Дублирование

- Данные контрагентов/договоров **дублируют** сценарии **финансов** (`openEntity`, карточка и т.д.).

### Итог и приоритеты

- **P0:** унификация **всех** запросов к `/api/*` через `API.*` **или** центральный `authFetch` с Bearer из `localStorage` — и быстрая ручная проверка: загружаются ли `/api/accounts` и `/api/settings/finance` в Network.
- **P1:** уменьшение размера `sales.js` по поддоменам (заказ / отгрузка / справочники).

- Мало; модулистая структура `references.js` (табы) — смотреть неиспользуемые `switch*Tab` при удалении сущностей.

### Итог и приоритеты

- **P1:** схема «какой справочник кто читает» в документации.
- **P2:** разбиение `dictionaries.js` на файлы по домену (как мини-роуты).

## 9. Реестр документов (`docs-registry-mod`)
