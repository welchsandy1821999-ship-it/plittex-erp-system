# 📜 Главные Заповеди ИИ-Ассистента (ERP PLITTEX)

Эти правила имеют наивысший приоритет. Любые действия ИИ должны сверяться с этим списком.

**1. Интроспекция и Первоисточник Истины (Self-Healing Docs)**
Никаких догадок. Схему БД читай через встроенный SQL-инструмент. Если база недоступна — смотри папку `migrations/`. Файлы `erp_*.md` используй как общую карту, но если найдешь расхождение — автоматически обнови Markdown-файл документации. 
**Карантин миграций:** Любые изменения схемы БД (`ALTER TABLE`, создание таблиц) производятся ИСКЛЮЧИТЕЛЬНО путем создания миграций в папке `migrations/`. Изменять структуру БД внутри API-маршрутов запрещено.

**2. Двухфазный контракт (Autonomous Edition)**
Анализ -> Утверждение -> Автономное исполнение.
Сначала обсуждаем проблему и план действий. После слова "Делай", ИИ выполняет ВЕСЬ объем работы автономно.
*Fast-Track:* Для локальных мелких правок (косметика, опечатки) план не требуется — выполняй сразу.
*Эскалация:* Если в процессе работы найден критический блокер, не позволяющий завершить задачу, правило "без остановок" нарушается — необходимо остановиться и показать проблему пользователю.

**3. Zero-Terminal Policy и Метод "Scratch Scripts"**
Строго запрещено использовать терминал для статического поиска (он виснет). Используй `grep_search`/`view_file`.
*Scratch Scripts:* Внедрять временные маршруты в `web.js` (Stealth API) — запрещено (это ломает nodemon). Если нужно запустить скрипт, собрать данные или восстановить БД — создай временный js-файл (например, в папке `scripts/`), запусти его напрямую через `node` (всегда обеспечив `process.exit(0)`) и затем удали.

**4. Связка данных (Тройной контроль)**
Всегда проверяй соответствие названий переменных и логики на всех трех уровнях системы: **База данных (SQL/ migrations)** -> **Бэкенд (роуты)** -> **Фронтенд (EJS/JS запросы)**. Например, если в базе склад называется `type='reserve'`, на фронтенде нельзя жестко проверять `id === 7`.

**5. Стандарт UI (Flex и Zero Inline Styles)**
Категорический запрет на инлайн-стили (`style="..."`). Вся вёрстка строится на семантических классах и Flexbox (`flex-row`, `gap-20`). Стили выносятся строго в `public/css/modules.css` (или `style.css`). При массовом UI-рефакторинге приветствуется автономная проверка интерфейса.

**6. Финансовая и Складская Точность (Big.js)**
Любые вычисления денег или складских остатков на фронтенде и в бэкенд-маршрутах ВСЕГДА должны оборачиваться в библиотеку `Big.js` (или `Math.round/parseFloat` с жесткой защитой). Игнорирование этого приведет к ошибкам плавающей точки (`0.1 + 0.2`).

**7. Auto-Run Protocol (Commander's Trust)**
Используй `SafeToAutoRun: true` для всех команд терминала, которые не являются деструктивными, чтобы не отвлекать пользователя кликами "Run". (Сборки, рестарты серверов, линтеры).

**8. Сквозной фокус**
Мы занимаемся ровно одной задачей до тех пор, пока пользователь прямо не скажет: "Мы закончили, идем дальше". До этого момента 100% внимания ИИ сосредоточено на доведении текущей задачи до идеала. Не забегать вперед, не предлагать переключения на другие модули.

**9. Git-снимок перед изменениями (Mandatory Checkpoint)**
Перед каждым изменением кода ИИ ОБЯЗАН сделать `git add -A; git commit -m "checkpoint: <описание>"`. Это создает точку восстановления на случай ошибки. Без коммита — код не трогаем.

**10. Дизайн-код ссылок (Anti-Underline)**
Отмена подчеркивания ссылок: Категорически запрещено добавлять подчеркивание к ссылкам (ни через style, ни через глобальные CSS-правила для a, ни через классы вроде .underline). Ссылки в ERP не подчеркиваются. Использовать text-decoration: none или класс .text-decoration-none.
