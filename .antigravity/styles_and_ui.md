# Стандарты верстки и UI (ERP PLITTEX)

Цель: поддерживаемая верстка без инлайн-мусора, с единым набором паттернов компонентов и сеток.

## 1) Базовые принципы
- `style="..."` в UI-шаблонах запрещен. Все стили выносить в CSS.
- Исключение: печатные HTML/PDF-генераторы, где inline допустим для стабильной печати (см. `.cursorrules`).
- Глобальные стили и утилиты: `public/css/style.css`.
- Модульные стили: `public/css/modules.css`.

## 2) Сетки и компоновка
- Использовать тот layout-паттерн, который уже принят в модуле: CSS Grid, Flex или их комбинацию.
- Для форм следовать существующим оберткам (`form-grid`, `form-group`, `col-span-*`, `input-modern`), не внедрять новый стиль локально.
- Для горизонтальных компоновок использовать utility-классы (`flex-row`, `flex-between`, `align-center`, `gap-*`, `flex-wrap`) вместо ad-hoc правил.

## 3) Дизайн-токены и компоненты
- Цвета только через CSS variables (`--surface`, `--surface-alt`, `--text-main`, `--text-muted`, `--primary`, `--success`, `--danger`, `--border`).
- Базовые элементы формы и кнопки должны использовать проектные классы (`input-modern`, `btn`, `btn-blue`, `btn-outline`).
- Высоты и отступы выравнивать под существующий UI-модуль, а не через локальные числовые хаки.

## 4) Управление видимостью и состояниями
- Не управлять видимостью через `element.style.display = ...` без причины.
- Предпочитать классы состояния (`inv-hidden`, `d-none` и модульные hidden-классы) и `classList`.
- Если для блока нужна “скрыто по умолчанию” логика, определить ее в CSS-классе и переключать только класс.

## 5) Контроль качества UI-рефакторинга
- Перед изменением: зафиксировать текущий визуальный паттерн в модуле.
- После изменения: проверить выравнивание, адаптивность, перенос заголовков и единообразие высоты полей.
- Если визуальная проверка недоступна, передать пользователю короткий чеклист сценариев проверки.
