# 🗄️ Протокол Управления Данными (ERP PLITTEX)

Документ описывает строгие правила для ИИ-ассистента при работе с PostgreSQL, API-маршрутами и моделями данных. Никаких догадок и предположений.

## 1. Взаимодействие с базой данных на Бэкенде (routes)
- **Защита транзакций:** Любые мутирующие операции (INSERT, UPDATE, DELETE из нескольких таблиц) ДОЛЖНЫ быть обернуты в `withTransaction(pool, async (client) => { ... })`. Использование ручных `BEGIN/COMMIT` строго запрещено во избежание утечек коннектов.
- **Математика:** Финансовые расчёты на сервере и клиенте должны производиться с помощью библиотеки `Big.js` для предотвращения ошибок плавающей точки.
- **Идентификаторы Складов (Warehouse ID):** Запрещено хардкодить `warehouse_id = 7` (Резерв) или `4/5` (Продукция/Уценка). Склады ищутся строго через функцию `await getWhId(client, 'warehouse_type')`.

## 2. Работа со Складом (Inventory)
- **Единый источник:** В системе НЕТ таблицы текущих остатков. Все остатки – это агрегация `SUM(quantity)` из таблицы `inventory_movements`.
- **Резервирование:** Резерв – это физическое перемещение товара в `inventory_movements` на виртуальный склад типа `reserve`. При расчете "Свободного" товара (На складе), количество на складе `finished` УЖЕ не содержит зарезервированный товар. Вычитать резерв из остатков готовой продукции повторно — ошибка.

## 3. Целостность (Foreign Keys & Deletion)
- **Soft Delete:** Использовать флаги логического удаления (например, `is_deleted = true` в `items` и `transactions`, `status = 'deleted'` в `employees`).
- **Блокировка CRUD:** Hard Delete допустим только для сущностей без каскадных связей. Использовать Cascade Delete с крайней осторожностью (см. AUDIT-003 для counterparties).

## 4. Общие принципы API
- Изменения маршрутов, предполагающие мутацию данных, должны (в большинстве случаев) заворачиваться в middleware `requireAdmin`, если только это не открытый API (согласно аудиту безопасности).
