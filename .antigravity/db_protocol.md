# 🗄️ Протокол Управления Данными (ERP PLITTEX)

Документ регламентирует строгие правила для ИИ при работе с PostgreSQL, API-маршрутами и моделями данных. Никаких догадок и предположений.

## 1. Запрет на прямые изменения (Карантин миграций)
Любые изменения схемы БД (`ALTER TABLE`, создание новых таблиц, триггеров) производятся ИСКЛЮЧИТЕЛЬНО путем добавления новых миграций в папку `migrations/`. Изменять структуру БД внутри API-маршрутов или напрямую в консоли (без создания `.sql` файла-бэкапа) СТРОГО запрещено.

## 2. Взаимодействие с базой данных на Бэкенде (routes)
- **Защита транзакций:** Любые мутирующие операции (INSERT, UPDATE, DELETE из нескольких взаимосвязанных таблиц) ДОЛЖНЫ быть обернуты в `withTransaction(pool, async (client) => { ... })`. Использование ручных `BEGIN/COMMIT` в маршрутах запрещено, чтобы не допускать утечек коннектов.
- **Математика:** Финансовые расчёты на сервере и клиенте должны производиться с помощью библиотеки `Big.js` для предотвращения ошибок плавающей точки.
- **Идентификаторы Складов (Warehouse ID):** Запрещено хардкодить `warehouse_id = 7` (Резерв) или `4/5` (Продукция/Уценка). Склады ищутся строго через функцию `await getWhId(client, 'warehouse_type')`.

## 3. Работа со Складом (Inventory)
- **Единый источник:** В системе НЕТ отдельной таблицы текущих остатков. Все остатки – это агрегация `SUM(quantity)` из таблицы `inventory_movements`.
- **Резервирование:** Резерв – это физическое перемещение товара в `inventory_movements` на виртуальный склад типа `reserve`. При расчете "Свободного" товара (На складе), количество на складе `finished` УЖЕ не содержит зарезервированный товар. Вычитать резерв из остатков готовой продукции повторно — ошибка.

## 4. Целостность (Foreign Keys & Deletion)
- **Soft Delete:** Использовать флаги логического удаления (например, `is_deleted = true` в `items` и `transactions`, `status = 'deleted'` в `employees`).
- **Блокировка CRUD:** Hard Delete допустим только для сущностей без каскадных связей. Использовать Cascade Delete с осторожностью.

## 5. Синхронизация слоев и документации
- Любое изменение бизнес-логики проверять по цепочке `SQL -> routes -> public/js -> views/modules`.
- При выявлении расхождений между кодом и docs, обновлять документацию в той же задаче.
- Источник архитектурного контекста: `erp_architecture_tree.md` и `erp_technical_docs.md`.

## 6. Практика проверок перед merge
- Для мутаций с финансовым эффектом проверять идемпотентность и корректность отката транзакций.
- Не допускать “магических” SQL-правок в рантайме; все структурные изменения держать в миграционном процессе проекта.
