## 🛡 Архитектура авторизации

```
web.js:195 → app.use('/api', authenticateToken)  ← ГЛОБАЛЬНЫЙ JWT для ВСЕХ /api/* маршрутов
web.js:198-205 → app.use('/', <routes>)          ← Маршруты подключены ПОСЛЕ middleware
```

> **Ключевое открытие**: Глобальный `authenticateToken` в `web.js:195` покрывает **ВСЕ** `/api/*` маршруты JWT-токеном.  
> Маршруты вне `/api/` (например `/print/*`) требуют индивидуального middleware.

В предыдущей версии отчёта ~70+ таких маршрутов были ошибочно помечены как «открытые» — это **ложные срабатывания**, удалённые из данной версии.

## 🚨 Очередь на исправление (Критичные Баги)

Только подтверждённые логические дефекты, влияющие на **целостность данных** и **стабильность системы**. Приоритет: от наиболее критичного к менее.

### BUG-001: 🔴 Отсутствие `UPDATE accounts` при закупке сырья

| Поле | Значение |
|------|----------|
| **Файл** | `routes/inventory.js`, строки 1272-1277 |
| **Процесс** | `POST /api/inventory/purchase` |
| **Суть** | После INSERT expense-транзакции **нигде не выполняется** `UPDATE accounts SET balance = ...`. Баланс кассы не уменьшается. |
| **Контраст** | `POST /api/sales/checkout` (sales.js:393-402) — **пересчитывает** баланс. `DELETE /api/transactions/:id` (finance.js:1240) — **пересчитывает** все балансы. |
| **Воспроизведение** | Закупить сырьё на 100 000 ₽ с оплатой через кассу → Перейти в Финансы → Баланс кассы **не изменился**. |
| **Влияние** | Завышенный баланс кассы. Рассинхрон между `transactions` и `accounts.balance`. |
| **Решение** | Добавить стандартный `UPDATE accounts a SET balance = ROUND(COALESCE((SELECT SUM(...) FROM transactions t WHERE t.account_id = a.id ...), 0), 2) WHERE a.id = $1` после INSERT transaction. |

---

### BUG-002: 🔴 Отсутствие `SELECT FOR UPDATE` при отгрузке

| Поле | Значение |
|------|----------|
| **Файл** | `routes/sales.js`, строки 445-449 |
| **Процесс** | `POST /api/sales/orders/:id/ship` |
| **Суть** | Нет блокировки строк `client_order_items`, нет проверки `qty_shipped + qty ≤ qty_ordered`. Два менеджера могут одновременно отгрузить один и тот же заказ → суммарная отгрузка > 100%. |
| **Дополнительно** | Нет проверки баланса reserve-склада перед списанием. При ручной отмене резерва → минусовые остатки. |
| **Влияние** | Race condition → двойная отгрузка, минусовые остатки на складе-резерве. |
| **Решение** | Добавить `SELECT qty_ordered, qty_shipped FROM client_order_items WHERE id = $1 FOR UPDATE`, затем проверку `qty_shipped + newQty ≤ qty_ordered`. |

### BUG-003: 🔴 Распалубка — нет серверной проверки остатка в сушилке

| Поле | Значение |
|------|----------|
| **Файл** | `routes/inventory.js`, строка 866 |
| **Процесс** | `POST /api/move-wip` |
| **Суть** | `currentWipQty` берётся из тела запроса. Нет `SELECT SUM(quantity) FROM inventory_movements WHERE batch_id = $1 AND warehouse_id = drying FOR UPDATE`. Два оператора → двойное списание. |
| **Дополнительно** | При `isComplete = true` и частичном выходе (good + grade2 + scrap < currentWipQty) — разница **бесследно исчезает** (нет записи `loss`/`writeoff`). |
| **Влияние** | Race condition → минусовые остатки. «Чёрная дыра» при частичном закрытии. |
| **Решение** | Серверная проверка реального баланса + генерация записи `writeoff` для разницы. |

## 🏗️ Архитектурные Решения (Бизнес-правила)

Следующие находки были первоначально классифицированы как уязвимости, но являются **осознанными бизнес-решениями**, обеспечивающими гибкость для отделов продаж и производства.

### AR-001: Цена товара задаётся менеджером на фронтенде (Checkout)

| Поле | Значение |
|------|----------|
| **Файлы** | `sales.js:1407` (фронт), `sales.js:237` (бэкенд) |
| **Поведение** | Менеджер устанавливает цену при добавлении товара в корзину. Сервер принимает эту цену без перезапроса из `items.current_price`. |
| **Бизнес-обоснование** | Менеджеры продаж имеют право давать **индивидуальные скидки** и устанавливать **договорные цены**, отличающиеся от прайса. Жёсткая привязка к каталогу сломает процесс B2B-продаж. |
| **Рекомендация** | Добавить **логирование**: записывать разницу между `item.current_price` и `payload.price` в поле `price_deviation` для аналитики и контроля скидок. Не блокировать. |

### AR-002: Расход сырья задаётся мастером на фронтенде (Фиксация смены)

| Поле | Значение |
|------|----------|
| **Файлы** | `production.js:510-522` (фронт), `production.js:302` (бэкенд) |
| **Поведение** | Мастер смены вводит фактический расход сырья (замесы × норма на замес). Сервер проверяет наличие на складе, но не проверяет соответствие рецептуре. |
| **Бизнес-обоснование** | Рецептуры — это **идеальные нормы**. В реальности каждый замес отличается: влажность песка, температура, плотность пластификатора. Мастер **обязан** указывать фактический расход, а не расчётный. |
| **Рекомендация** | Добавить **предупреждение** (не блокировку): если фактический расход отклоняется от рецептуры > 20%, выводить предупреждение в UI. Не отклонять. |

### AR-003: `totalCost` при закупке задаётся вручную

| Поле | Значение |
|------|----------|
| **Файлы** | `purchases.js:186` (фронт), `inventory.js:1240` (бэкенд) |
| **Поведение** | Менеджер может вручную отредактировать поле «Итого за сырьё». Бэкенд **приоритетно** использует `frontendTotal`, а не `qty × price`. |
| **Бизнес-обоснование** | Поставщик может дать **оптовую скидку на партию** (фиксированная цена за всю поставку), которая не раскладывается ровно на цену за единицу. Ручной ввод итоговой суммы — стандартная практика закупочного учёта. |
| **Рекомендация** | Сохранять и `totalCost`, и `qty × price` в отдельных полях для аудита. Нынешняя логика корректна для бизнеса. |

### AR-004: Скидка при checkout — бизнес-диапазон

| Поле | Значение |
|------|----------|
| **Файлы** | `sales.js:242` (бэкенд) |
| **Поведение** | Бэкенд не ограничивает `discount` сверху (теоретически можно передать > 100%). |
| **Рекомендация** | Добавить серверную проверку `if (discount < 0 || discount > 100) throw new Error('Недопустимый размер скидки')`. Это **не бизнес-правило, а баг** — см. BUG-009 ниже. |

> **BUG-009** (🟢 LOW): Серверу следует ограничить `discount` значениями 0–100%. Сейчас `discount = 110` создаст заказ с отрицательной суммой. Фронтенд ограничивает ввод, но сервер — нет.

## 📝 Технический долг (Низкий приоритет)

Позиции, не требующие срочного исправления, но заслуживающие внимания при рефакторинге.

### 1. Утечка стек-трейсов

~50+ мест в routes/ возвращают `res.status(500).json({ error: err.message })`. В продакшене следует заменить на generic-сообщение и логировать через `console.error`.

### 2. Stored XSS (innerHTML)

~20+ мест в `public/js/` используют `innerHTML` для рендеринга данных. В `core.js` есть `Utils.escapeHtml`, но он **почти нигде не применяется**. Риск Stored XSS при внедрении скрипта в `name`/`description` контрагента.

### 3. Утечки памяти (addEventListener без removeEventListener)

SPA-архитектура (`switchModule`) переключает модули без очистки обработчиков событий. Файлы: `dashboard.js`, `docs_registry.js`, `production.js`, `references.js`.

### 4. SELECT * без LIMIT

~9 мест в routes/ используют `SELECT *` без пагинации. При росте объёмов данных — деградация производительности.

### 5. N+1 запросы

Обнаружены `await pool.query` внутри циклов `for` в файлах: `dictionaries.js`, `finance.js`, `hr.js`, `inventory.js`, `production.js`, `sales.js`. Следует заменить на batch-запросы (`WHERE id = ANY($1::int[])`).

### 6. Ручное управление транзакциями

`finance.js → POST /api/invoices`: Использует `pool.connect()` + `BEGIN/COMMIT/ROLLBACK` вместо `withTransaction()`. Риск утечки пула соединений при не-вызове `client.release()`.

### 7. Синхронные файловые операции

`docs.js:456`: `fs.existsSync` + `fs.mkdirSync` — блокируют Event Loop. Заменить на `fs.promises.mkdir(..., { recursive: true })`.

### 8. `parseFloat` вместо `Big.js` в финансовых агрегатах

`sales.js:268` (формула баланса клиента для offset): Использует `parseFloat` для крупных финансовых агрегатов. При суммах > 1M₽ возможны артефакты плавающей точки.

### 9. Несогласованная точность

Производственный модуль записывает `quantity` с `toFixed(4)`, все остальные — с `toFixed(2)`. При агрегации `ROUND(quantity, 2)` в SQL — обрезка 4-значных дробей.
